# PR 评分系统说明

## 评分标准 (0-100分)

PR 的质量评分基于以下维度：

### 1. CI 状态 (30分)

- **SUCCESS**: 30分 ✅
- **PENDING**: 15分 ⏳
- **FAILURE**: 0分 ❌
- **无检查**: 0分 ⚪

### 2. 代码规模 (20分)

合理的代码量获得最高分，过小或过大都会扣分：

- **200-800 行变更**: 20分 🎯
- **<200 行**: 15分（小但可用）
- **800-1500 行**: 10分（较大但可管理）
- **>1500 行**: 5分（过大）

### 3. 修改聚焦度 (15分)

修改的文件数越少，PR 越聚焦：

- **≤3 个文件**: 15分 🎯
- **4-5 个文件**: 12分
- **6-10 个文件**: 8分
- **>10 个文件**: 4分

### 4. 审核情况 (15分)

收到更多的 approval 表示 PR 质量更高：

- **≥2 个 approvals**: 15分 🏆
- **1 个 approval**: 10分
- **无 approval**: 0分

### 5. 文档完整性 (10分)

- **有 PR 描述**: 5分
- **有 documentation 标签**: 5分
- **总计**: 10分

### 6. 时效性 (10分)

新提交的 PR 优先级更高：

- **≤1 天**: 10分
- **2-3 天**: 7分
- **4-7 天**: 4分
- **>7 天**: 2分

## 评分等级

- **🏆 80-100分**: 优秀 - 高质量 PR，值得优先审核
- **⭐ 60-79分**: 良好 - 质量不错，可以审核
- **👍 40-59分**: 一般 - 基本达标，但有待改进
- **💪 0-39分**: 需改进 - 建议优化后再提交

## 优化建议

### 提高评分的方法

1. **确保 CI 通过** (最重要，30分)
   - 修复所有失败的检查
   - 确保测试通过

2. **控制 PR 规模**
   - 合理的代码量（200-800行）
   - 聚焦在单一功能
   - 大改动拆分成多个小 PR

3. **减少修改文件数** (15分)
   - 一次只改必要的内容
   - 避免重构+功能混在一起

4. **获取 Approvals** (15分)
   - 主动请求审核
   - 响应 review 意见

5. **完善文档** (10分)
   - 写详细的 PR 描述
   - 说明变更原因和影响
   - 添加 `documentation` 标签

6. **保持及时性** (10分)
   - 及时创建 PR
   - 避免长时间堆积

## 示例

### 高分 PR (85分)

```
✨ PR #42: 实现用户认证功能 ⭐ 85/100
👤 作者: @zhangnan27
⏱️ 创建: 2小时前
📊 变更: +450/-120 (5个文件)
✅ CI: 通过
👥 等待审核

评分详情:
- CI状态: 30/30 (SUCCESS)
- 代码规模: 20/20 (570行，合理)
- 修改聚焦: 12/15 (5个文件)
- 审核情况: 10/15 (1个approval)
- 文档完整: 10/10 (有描述)
- 时效性: 10/10 (2小时内)
```

### 低分 PR (25分)

```
⚠️ PR #15: 修复登录 bug 💪 25/100
👤 作者: @junior-dev
⏱️ 创建: 3天前
📊 变更: +2000/-50 (15个文件)
❌ CI: 失败
👥 等待审核

评分详情:
- CI状态: 0/30 (FAILURE)
- 代码规模: 5/20 (2050行，过大)
- 修改聚焦: 4/15 (15个文件)
- 审核情况: 0/15 (无approval)
- 文档完整: 5/10 (有描述)
- 时效性: 7/10 (3天内)

改进建议:
- 修复 CI 失败
- 考虑拆分成多个小 PR
- 主动请求审核
```

## 使用场景

1. **优先审核排序**: 高分 PR 优先审核
2. **质量趋势**: 追踪 PR 质量变化
3. **贡献者反馈**: 帮助贡献者改进
4. **团队指标**: 整体 PR 质量统计
